# T-029-02 Review: Proposal Mock Trait

## Summary

Created a trait abstraction (`ProposalNarrativeGenerator`) for proposal narrative generation in `pt-proposal`, with three implementations: `BamlProposalGenerator` (real LLM via BAML), `MockProposalGenerator` (deterministic canned narratives), and `ClaudeCliGenerator` (local `claude` CLI, auto-generated by hook). Integrated the trait into `AppState` via `Arc<dyn ProposalNarrativeGenerator>` and updated all test helpers to use the mock.

## Open Concerns

1. **Smoke test documentation**: The ticket asks for `ANTHROPIC_API_KEY=... just test-smoke` documentation. No smoke test recipe exists yet. Recommend adding an `#[ignore = "requires ANTHROPIC_API_KEY"]` test in a follow-up, and a `just test-smoke` recipe.

2. **`ClaudeCliGenerator` (hook-generated)**: Auto-generated by a hook. Uses `B.GenerateProposalNarrative.parse()` which assumes BAML has a `parse` method on the function struct. This compiles but hasn't been runtime-tested. If the `parse()` method doesn't exist at runtime (or behaves unexpectedly), this impl would fail. Low risk since it's an alternative impl, not the default.

3. **BAML types lack serde**: `ProposalContent` and friends derive `BamlEncode`/`BamlDecode` but not `Serialize`/`Deserialize`. If a future route needs to return `ProposalContent` as JSON, a DTO conversion layer will be needed. This is expected and not a blocker.
