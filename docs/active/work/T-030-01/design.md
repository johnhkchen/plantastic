# T-030-01 Design — Typst Template & PDF Rendering Pipeline

## Decision 1: Typst integration approach

### Options
**A. Raw `typst` + `typst-pdf`** — Implement `World` trait manually (~100 lines boilerplate for font loading, file resolution, caching).
**B. `typst-as-lib`** — Community wrapper, handles World boilerplate. Actively maintained (v0.15.4 tracks typst 0.14.x).
**C. `typst-bake`** — Build-time embedding of templates, fonts, packages into binary.

### Decision: B — `typst-as-lib`
- Eliminates World trait boilerplate (font discovery, source caching, file resolution)
- Template content can be embedded via `include_str!` at compile time
- JSON data via `sys.inputs` is first-class
- If it becomes limiting, migrating to raw typst is straightforward (same compile/pdf API)
- Rejected A: unnecessary boilerplate for our single-template use case
- Rejected C: too opinionated, less control over runtime behavior

## Decision 2: ProposalDocument struct location

### Decision: Define in pt-proposal
The struct merges types from pt-quote, BAML, and branding. It belongs in pt-proposal because:
- It's the input to `render_proposal()`, which lives in pt-proposal
- pt-proposal already depends on serde; adding pt-quote dependency is natural
- `TenantBranding` is a new struct (not TenantRow — decoupled from DB schema)

```rust
pub struct TenantBranding {
    pub company_name: String,
    pub logo_url: Option<String>,
    pub primary_color: Option<String>,  // hex, e.g. "#4CAF50"
    pub phone: Option<String>,
    pub email: Option<String>,
}

pub struct ProposalDocument {
    pub project_name: String,
    pub project_address: String,
    pub date: String,                    // pre-formatted date string
    pub branding: TenantBranding,
    pub narrative: ProposalContent,      // from BAML
    pub good_quote: Quote,              // from pt-quote
    pub better_quote: Quote,
    pub best_quote: Quote,
}
```

## Decision 3: Template embedding

### Decision: `include_str!` at compile time
- Template is `templates/proposal.typ` relative to crate root
- Embedded into binary via `include_str!("../templates/proposal.typ")`
- No filesystem access needed at runtime → Lambda-friendly
- Template changes require recompile — acceptable for our release cadence

## Decision 4: Dollar amount formatting

### Decision: Pre-format in Rust, pass as strings
- Decimal fields serialize as strings via `serde-with-str` already
- Add a `TemplateData` intermediate that converts Decimals to formatted dollar strings
- Template receives `"$1,530.00"` not `"1530.00"` — no formatting logic in Typst
- Keeps the template simple and ensures consistency with pt-quote precision

## Decision 5: Font strategy

### Decision: Use Typst's built-in fonts
- typst-as-lib bundles default fonts (New Computer Modern, etc.)
- Professional enough for MVP; brand fonts can be added later
- No extra binary size management needed

## Decision 6: Template layout

### Sections (top to bottom)
1. **Header**: Company name (large), date (right-aligned), logo placeholder
2. **Project info**: Project name, address
3. **Intro paragraph**: From `narrative.intro_paragraph`
4. **Three-tier comparison table**: The centerpiece
   - Columns: Zone | Good | Better | Best
   - Rows: one per zone, showing materials + line totals
   - Subtotal row, tax row (if any), grand total row
   - Color-code tier headers with branding primary_color
5. **Tier narratives**: One section per tier with headline + description + differentiators
6. **Zone callouts**: From `narrative.zone_callouts`
7. **Closing paragraph + CTA**: From `narrative.closing_paragraph`
8. **Footer**: Page numbers, "Generated by Plantastic"

### Data transformation pipeline
```
ProposalDocument
    → serde_json::to_string() → TemplateData (with formatted dollars)
    → sys.inputs.data
    → Template parses via json(bytes(sys.inputs.data))
    → Typst layout → compile → PDF bytes
```
