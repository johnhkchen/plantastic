# T-035-01 Research: BAML AnalyzePlanView

## Objective

Add an `AnalyzePlanView` BAML function that takes an annotated plan-view PNG (with feature labels), lot dimensions, address, and classified features, and returns a `SiteAnalysis` with features, suggested zones, and site observations. This is the product's shift from "scan processor" to "AI landscape designer."

## Existing BAML Infrastructure

### BAML Source Files (`baml_src/`)

| File | Function | Input | Output |
|------|----------|-------|--------|
| `classify.baml` | `ClassifyFeatures` | candidates[], address, climate_zone | ClassifiedFeature[] |
| `planter.baml` | `EstimatePlanter` | gap dimensions, features, climate, address | PlanterEstimate (styles[]) |
| `proposal.baml` | `GenerateProposalNarrative` | company, project, address, tiers[] | ProposalContent |
| `clients.baml` | Client definitions | — | ProposalFallback (Haiku → Sonnet) |
| `generators.baml` | Rust codegen config | — | baml_client/ output |

**Key patterns:**
- All functions use `client ProposalFallback` (Haiku first, Sonnet fallback)
- `{{ ctx.output_format }}` injects the JSON schema into prompts
- BAML `test` blocks define fixtures for the BAML playground
- Types are defined inline in each `.baml` file

### Generated Client (`baml_client/`)

Auto-generated by `baml-cli generate`. Key files:
- `functions/async_client.rs`: `baml_function_async!` macro generates per-function structs with `call()`, `stream()`, `parse()`, `parse_stream()` methods. `BamlAsyncClient` struct aggregates all functions as fields. Static singleton `B`.
- `types/classes.rs`: Generated Rust structs with `BamlEncode`/`BamlDecode` derives
- `types/mod.rs`: `Types` enum for compile-time type registry, re-exports `baml::Image`

**Image support:** `baml::Image` type is re-exported in `types/mod.rs`. BAML supports `image` type natively — the runtime handles base64 encoding/decoding.

### Three-Tier Crate Pattern

Both `pt-features` and `pt-planter` follow identical structure:

```
crates/pt-{name}/
  Cargo.toml          # workspace deps + pt-scan dep
  src/
    lib.rs            # #[path] re-include of baml_client, re-exports
    {trait}.rs         # Trait + BamlXxx implementation
    claude_cli.rs     # ClaudeCliXxx — subscription-based fallback
    mock.rs           # MockXxx + fixture function
    error.rs          # thiserror enum: {Name}(String), InvalidInput(String)
```

**Pattern details:**
1. `lib.rs` includes `baml_client` via `#[path = "../../../baml_client/mod.rs"] mod baml_client;` with broad `#[allow(...)]` suppression
2. Trait has single async method returning `Result<GeneratedType, CrateError>`
3. `BamlXxx` calls `crate::B.FunctionName.call(...)` directly
4. `ClaudeCliXxx` builds a prompt string, calls `claude -p ... --output-format text`, then uses `crate::B.FunctionName.parse(&raw)` to parse
5. `MockXxx` returns a hand-crafted fixture (realistic SF Bay Area data)
6. `MockFailingXxx` always returns an error (for error-path tests)
7. Re-exports: generated types, `B` singleton, function struct, trait, implementations, errors

### pt-scan Annotated Plan View

`crates/pt-scan/src/annotate.rs` provides:
- `FeatureAnnotation` struct: label, category, bbox, centroid
- `annotate_plan_view_png()`: overlays classified features on base plan-view PNG
- Category-colored bounding boxes, labeled with confidence scores
- The annotated PNG is the input for AnalyzePlanView

`crates/pt-scan/src/export.rs` generates:
- `TerrainOutput`: mesh_glb, plan_view_png, metadata
- Plan view is orthographic top-down PNG

### Image Handling in BAML

BAML supports `image` as a native type. In the prompt, images can be referenced as `{{ image_var }}` and the runtime handles multimodal API calls. The `baml::Image` type supports construction from base64 data via `Image::from_base64("media_type", base64_string)`.

Looking at `baml_client/types/mod.rs` line 19: `pub use baml::{Audio, Image, Pdf, Video};` — these types are already available.

## Dependencies

T-034-02 (annotated plan view) is listed as a dependency and appears to be in progress — the annotated plan view PNG generation and the `FeatureAnnotation` types already exist in `pt-scan/src/annotate.rs`.

## Constraints

1. BAML client regeneration: adding a new `.baml` file requires `baml-cli generate` to update `baml_client/`
2. The new crate must follow the exact same pattern as pt-features/pt-planter for consistency
3. Mock fixture must use realistic Powell & Market data (same scenario as other mocks)
4. `just check` must pass (fmt, lint, test, scenarios)
5. No real LLM calls in tests — mock trait only
6. ClassifiedFeature is defined in `classify.baml` and shared — AnalyzePlanView can reference it

## Open Questions

1. Should `AnalyzePlanView` accept `image` (BAML native type) or `string` (base64)? The BAML `image` type is more idiomatic and handles multimodal API calls natively.
2. Should the new crate be `pt-analyzer` or `pt-site-analysis`? Following the terse naming of `pt-features`, `pt-planter`, `pt-analyzer` is more consistent.
3. Does the scenario test for this capability already exist, or do we need to add one? No existing scenario covers site analysis — may need S.2.5 or similar.
