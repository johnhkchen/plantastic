# T-013-02 Research: SvelteKit Iframe Bridge

## What exists

### Bevy viewer (apps/viewer/)

Standalone WASM app, **not** a workspace member (`exclude = ["apps/viewer"]` in root Cargo.toml).
Built with Trunk, outputs to `apps/viewer/dist/`.

**Source files:**
- `main.rs` — App entry. DefaultPlugins with `#bevy-canvas` selector, `fit_canvas_to_parent: true`, `prevent_default_event_handling: true`. Adds CameraPlugin, ScenePlugin, LightingPlugin, diagnostics.
- `scene.rs` — ScenePlugin. Loads `models/test_scene.glb` via AssetServer at startup. `track_scene_load` system polls `Assets<Gltf>`, spawns `SceneRoot` when loaded. Hardcoded asset path.
- `camera.rs` — PanOrbitCamera (bevy_panorbit_camera v0.34). Left-click orbit, right-click pan, scroll zoom, touch support. Position `(3.0, 3.0, 5.0)` looking at origin.
- `lighting.rs` — DirectionalLight (10k lux, shadows, -45° pitch, 30° yaw) + GlobalAmbientLight (200 brightness).

**Build output (dist/):**
- `index.html` — Generated by Trunk. Has wasm-bindgen init script, exposes `window.wasmBindings`, fires `TrunkApplicationStarted` event.
- `plantastic-viewer-*.js` (102 KB) — wasm-bindgen glue.
- `plantastic-viewer-*_bg.wasm` (14 MB raw, 10 MB after wasm-opt -Oz).
- `assets/models/test_scene.glb` (1.5 KB test model).

**Template (index.html):**
- Canvas `#bevy-canvas` fills parent. Loading spinner overlay.
- MutationObserver hides spinner when canvas gets dimensions.
- `data-wasm-opt="0"` — wasm-opt disabled in Trunk (Rust 1.84+ compat issue).
- `data-trunk rel="copy-dir" href="assets"` — copies assets to dist.

**No JS interop currently.** No postMessage handling, no `#[wasm_bindgen]` exports, no custom events beyond the Trunk-generated `TrunkApplicationStarted`.

### SvelteKit app (web/)

SvelteKit 2 + Svelte 5 runes. CF Pages adapter. Vite 7. Tailwind 4.

**Viewer route:** `web/src/routes/(app)/project/[id]/viewer/+page.svelte`
- Current content: `<h2>3D Viewer</h2><p>Coming soon.</p>`

**Project layout:** `web/src/routes/(app)/project/[id]/+layout.svelte`
- Loads project by ID from API. Renders `TabNav` + children.
- `TabNav.svelte` — 5 tabs: Editor, Materials, Quote, Viewer, Export.

**Layout structure:** `<div class="mx-auto max-w-5xl">` — constrained width container. The viewer will need to break out of this or the iframe won't get enough height.

**Existing component patterns:**
- ZoneEditor.svelte — Canvas-based, uses `$state`/`$effect`/`$derived` runes. Event dispatch via callback props (Svelte 5 style, not `createEventDispatcher`).
- All components use Tailwind utility classes.
- API client at `$lib/api/client.ts` — `apiFetch<T>(path, options)`.

**Project ID flow:** `/project/[id]/+layout.ts` exports `{ id: params.id }`. Child pages receive this via `LayoutData`.

### Bevy WASM ↔ JavaScript interop options

**1. wasm-bindgen `#[wasm_bindgen]` exports:**
- Expose Rust functions to JS. Trunk's generated JS already calls `init()`.
- Can expose functions like `load_scene(url: &str)`, `set_light_angle(degrees: f32)`.
- Problem: Bevy's ECS is not directly callable from JS. Need to use Bevy Events or Resources as the bridge — JS calls a `#[wasm_bindgen]` function that pushes an event, Bevy system reads the event.

**2. `web_sys::window().add_event_listener` (Custom DOM events):**
- Bevy system listens for CustomEvents on window.
- Host dispatches events, Bevy picks them up.
- Simpler than postMessage for same-origin, but postMessage works cross-origin (future-proof for CDN-served viewer).

**3. postMessage (iframe ↔ host):**
- Standard iframe communication. Works cross-origin.
- Viewer JS listens for `message` events on `window`.
- Host sends via `iframe.contentWindow.postMessage()`.
- Viewer sends back via `window.parent.postMessage()`.
- JSON payload — easy to version and extend.

**4. Shared channel via `MessageChannel`:**
- More complex, no real benefit over postMessage for this use case.

### Bevy picking / tap detection

**bevy_mod_picking** — The standard Bevy picking crate. But it's heavy and may not support WebGL2 well.

**Manual raycasting:** Bevy 0.18 has built-in `MeshRayCast` (formerly `bevy_mod_raycast`). Can cast a ray from screen coordinates through the camera, check intersection with meshes. This is the lighter approach.

**For this spike:** We need to detect taps on meshes and report the mesh name back to the host. A system that reads touch/mouse input, does a raycast, and finds the intersected entity's `Name` component is sufficient.

## Constraints and boundaries

1. **Viewer is excluded from workspace.** Cannot import workspace crates. All communication is via postMessage JSON.
2. **Trunk build.** Must integrate JS interop with Trunk's build pipeline. Trunk generates the JS glue — our postMessage listener goes in `index.html` or in Rust via `web_sys`.
3. **CF Pages deployment.** The viewer dist/ needs to be served somewhere. Options: (a) same CF Pages origin under `/viewer/`, (b) separate static hosting, (c) inline as static assets. For this spike, serving from same origin is simplest.
4. **iframe sizing.** The project layout uses `max-w-5xl` (max 1024px). The viewer page needs to render a tall iframe. A fixed aspect ratio container (16:9 or similar) is the standard pattern.
5. **Touch support.** Bevy has `touch` feature enabled. PanOrbitCamera supports touch. Tap detection needs to handle both mouse click and touch tap.
6. **Asset loading.** Currently loads from relative `assets/models/test_scene.glb`. The `loadScene(url)` command should allow loading from an absolute URL. Bevy's AssetServer can load from HTTP URLs in WASM via `asset_server.load("http://...")`.

## Assumptions to validate in Design

- Can Bevy's AssetServer load glTF from an arbitrary URL (not just relative paths)?
- Is `MeshRayCast` available in Bevy 0.18 with the features we have enabled?
- Can we add `web_sys` as a dependency to the viewer crate for postMessage handling?
- Will Trunk preserve custom `<script>` blocks in index.html alongside its generated code?
- What's the minimal Bevy feature set needed for raycasting?
